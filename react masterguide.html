Overview of React.js

React is a JavaScript library built by Facebook (now Meta) in 2013. But calling it just a "library" undersells what it really is ‚Äî it's a completely different way of thinking about building user interfaces.

Before React, building a dynamic web page meant this: when data changed, you'd hunt down the specific HTML element in the DOM, reach in, and update it manually. You were constantly juggling the state of your data and the state of your UI, keeping them in sync by hand. It was fragile, error-prone, and a nightmare to scale.

React's big idea was radical: describe what your UI should look like at any given moment, and let React figure out how to make it happen.

The Core Philosophy

React is declarative. You don't say "find the button, change its color to red." You say "when the user is logged in, the button IS red." React handles the rest.

Component-Based

Your entire UI is built from small, reusable, independent pieces called components. Each component manages its own logic and appearance, like LEGO bricks.

"React lets you build UI the same way you think ‚Äî in terms of what it should be, not how to get there."

Why React Over Plain JavaScript?

üéØ

Predictable UI Updates

When your data (state) changes, React automatically re-renders the right parts of the UI. No more manual DOM manipulation.

üß©

Reusable Components

Build a Button component once, use it 500 times across your app. Change it in one place, it updates everywhere.

‚ö°

Virtual DOM Performance

React doesn't touch the real DOM unless absolutely necessary. It uses a clever virtual copy to minimize expensive real DOM operations.

üåç

Massive Ecosystem

React has the largest ecosystem in frontend development. Routing, state management, UI libraries ‚Äî everything exists for React.

The Imperative vs Declarative Difference

This is the most important concept to grasp before anything else. Let's look at a real example:

‚ùå Imperative ‚Äî Old Way (Vanilla JS) 
// When a user logs in, I have to do ALL of this manually:
const handleLogin = (user) => {
  // 1. Find the element
  const btn = document.getElementById('login-btn');
  // 2. Change its text
  btn.textContent = `Hello, ${user.name}`;
  // 3. Change its style
  btn.style.background = 'green';
  // 4. Hide the login form
  document.getElementById('login-form').style.display = 'none';
  // 5. Show the dashboard
  document.getElementById('dashboard').style.display = 'block';
  // 6. Update the avatar
  document.getElementById('avatar').src = user.avatar;
  // You're managing 6 DOM operations. Miss one = broken UI.
};
‚úÖ Declarative ‚Äî React Way 
// Just describe what the UI LOOKS LIKE based on state.
// React figures out ALL the DOM changes automatically.
function App() {
  const [user, setUser] = useState(null);

  if (!user) {
    return <LoginForm onLogin={setUser} />;
  }

  return (
    <Dashboard
      user={user}
      greeting={`Hello, ${user.name}`}
    />
  );
  // React automatically shows LoginForm when user=null
  // and Dashboard when user is set. ZERO manual DOM work.
}
Key Insight

In React, you describe what the UI should look like given the current data. The moment the data changes, React re-renders and applies the minimum possible changes to the real DOM. You never touch the DOM directly.

The Virtual DOM

To understand the Virtual DOM, you first need to understand why it was invented ‚Äî and that means understanding the real DOM's biggest problem.

Why the Real DOM is Slow

üê¢ Real DOM Problem

The Real DOM (Document Object Model) is the browser's live representation of your HTML. Whenever you change it ‚Äî even one tiny attribute ‚Äî the browser may:

1. Recalculate styles for affected elements
2. Reflow (recalculate positions/sizes)
3. Repaint (redraw pixels on screen)

These operations are extremely expensive. In a complex app updating dozens of elements per second, this causes lag and jank.

‚ö° Virtual DOM Solution

React keeps a lightweight JavaScript copy of the DOM in memory ‚Äî the Virtual DOM. It's just a plain object tree, not real browser DOM nodes.

Operations on a JavaScript object are incredibly fast ‚Äî thousands of times faster than touching real DOM nodes.

React batches changes, computes the minimal diff, then makes exactly the right real DOM updates in one efficient pass.

How Virtual DOM Works: Step by Step

The Reconciliation Process
State Changes
‚Üí
New Virtual DOM Created
‚Üí
Diff Old vs New
(Diffing Algorithm)
‚Üí
Patch Real DOM
(Minimal Changes Only)

Step 1 ‚Äî State Changes

Imagine you have a counter displaying "5". The user clicks "+1". React knows state changed from 5 ‚Üí 6.

Step 2 ‚Äî New Virtual DOM Tree

React runs your component function again and creates a new Virtual DOM tree ‚Äî a fresh JavaScript object representing the new UI (with "6" in it).

Step 3 ‚Äî Diffing (Reconciliation)

React compares the OLD Virtual DOM tree with the NEW Virtual DOM tree using its diffing algorithm. It asks: "What changed?"

In our counter example, it finds: only the text node "5" ‚Üí "6" changed. Everything else is identical.

Step 4 ‚Äî Patch the Real DOM

React now touches the real DOM with surgical precision ‚Äî it changes only that one text node. Not the button, not the wrapper div, not the paragraph next to it. Just the exact character that changed.

Real-World Impact

In a Twitter-style feed with 500 posts, if you like one post, React changes only the heart icon's state on that one post. It doesn't re-render all 500 posts. Without Virtual DOM, naive implementations would touch or re-render far more.

Under The Hood

What does a Virtual DOM node actually look like?

It's just a plain JavaScript object. Nothing magical ‚Äî just a description of a DOM element:

Virtual DOM ‚Äî Plain JavaScript Object 
// When you write JSX like this:
<div className="card">
  <h1>Hello World</h1>
  <p>Welcome!</p>
</div>

// React internally converts it to this object:
{
  type: 'div',
  props: {
    className: 'card',
    children: [
      {
        type: 'h1',
        props: { children: 'Hello World' }
      },
      {
        type: 'p',
        props: { children: 'Welcome!' }
      }
    ]
  }
}
// Comparing two of these plain objects is FAST.
// Manipulating real DOM is SLOW. That's the whole trick.
Common Misconception

The Virtual DOM isn't always faster than directly manipulating the real DOM. In simple cases, direct DOM manipulation can be faster. The Virtual DOM's value is in making it easy to build complex apps without worrying about performance ‚Äî React optimizes for you automatically.

Components ‚Äî The Heart of React

Components are the fundamental building blocks of React. Everything you see on screen is a component or made of components. Think of them as smart HTML elements that can have their own data, logic, and appearance.

"A component is just a JavaScript function that returns what the UI should look like."

The Modern Way: Function Components

Today, all React components are written as functions. They receive data (called props) and return JSX (HTML-like syntax) describing the UI:

components/UserCard.jsx 
// A component is just a function with a capital first letter
function UserCard({ name, role, avatar }) {
  // JSX: looks like HTML but it's JavaScript!
  return (
    <div className="card">
      <img src={avatar} alt={name} />
      <h2>{name}</h2>   {/* {} lets you inject JS into JSX */}
      <p>{role}</p>
    </div>
  );
}

// Now use it like an HTML tag, anywhere in your app:
<UserCard
  name="Sarah Kim"
  role="Senior Engineer"
  avatar="/sarah.jpg"
/>

// Same component, different data = different card!
<UserCard
  name="Marcus Lee"
  role="Product Designer"
  avatar="/marcus.jpg"
/>

Understanding JSX

JSX stands for JavaScript XML. It's syntactic sugar that lets you write HTML-like code inside JavaScript. The browser doesn't understand JSX ‚Äî Babel (a tool) compiles it to plain React.createElement() calls:

What you write (JSX) 
<button
  className="btn"
  onClick={handleClick}
>
  Click Me!
</button>
What Babel compiles it to

  'button',
  {
    className: 'btn',
    onClick: handleClick
  },
  'Click Me!'

JSX Golden Rules

One Root Element

Every component must return a single root element. Use <div>, a Fragment <></>, or any wrapper.

2Ô∏è‚É£

className, not class

Since JSX is JavaScript, use className instead of class (class is a reserved JS keyword).

3Ô∏è‚É£

In HTML, <img> is fine. In JSX, you MUST close it: <img />. Every tag must be properly closed.

4Ô∏è‚É£

Inject any JavaScript expression using { }. Variables, ternaries, function calls ‚Äî all valid inside JSX.

Props ‚Äî Passing Data

Props: Talking to Components

Props (short for properties) are how you pass data from parent to child. They're like function arguments for your components. A component can never modify its own props ‚Äî they're read-only:

Props in Action 
// PARENT ‚Äî passes data as "attributes" to child
function App() {
  return (
    <Greeting
      name="Alex"
      age={25}                 {/* Numbers in {} */}
      isLoggedIn={true}         {/* Booleans in {} */}
      colors={['red', 'blue']}  {/* Arrays in {} */}
    />
  );
}

// CHILD ‚Äî receives props as a parameter object
function Greeting({ name, age, isLoggedIn, colors }) {
  return (
    <div>
      {/* Conditional rendering with ternary */}
      {isLoggedIn
        ? <p>Welcome back, {name}! You are {age} years old.</p>
        : <p>Please log in.</p>
      }
      {/* Rendering a list with .map() */}
      <ul>
        {colors.map(color =>
          <li key={color}>{color}</li>  {/* key is required for lists! */}
        )}
      </ul>
    </div>
  );
}
Props are Read-Only

A child component never modifies its own props. Data flows one direction: parent ‚Üí child. If a child needs to update data, it calls a function passed down as a prop (callback pattern). This one-directional flow is what makes React apps predictable and debuggable.

Component Lifecycle

Every React component has a lifecycle ‚Äî it's born (mounts), lives and updates (updates), and eventually dies (unmounts). Understanding this lifecycle is how you run code at the right time ‚Äî fetching data, setting up subscriptions, cleaning up timers.

Component Lifecycle Phases
Mount

Component appears in the DOM for the first time. Good time to: fetch initial data, set up subscriptions.

‚Üí
Update

State or props changed. Component re-renders. Good time to: respond to data changes, update external APIs.

‚Üí
Unmount

Component removed from DOM. Critical time to: clear timers, cancel subscriptions, prevent memory leaks.

The useEffect Hook ‚Äî Lifecycle in Modern React

In modern React (hooks era), lifecycle is handled with useEffect. It's one function that handles all three phases, controlled by its dependency array:

useEffect ‚Äî Complete Guide 
import { useState, useEffect } from 'react';

function DataFetcher({ userId }) {
  const [data, setData] = useState(null);

  // ‚îÄ‚îÄ 1. RUNS ONCE (on Mount) ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ
  // Empty array [] = "run this effect only when component mounts"
  useEffect(() => {
    console.log('Component mounted! Fetch initial data.');
    return () => console.log('Component unmounted! Cleanup here.');
  }, []);  // <-- Empty dependency array

  // ‚îÄ‚îÄ 2. RUNS ON UPDATE (when userId changes) ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ
  // [userId] = "re-run whenever userId changes"
  useEffect(() => {
    // Fetch data whenever userId changes
    fetch(`/api/users/${userId}`)
      .then(res => res.json())
      .then(data => setData(data));
  }, [userId]);  // <-- Re-runs when userId changes

  // ‚îÄ‚îÄ 3. RUNS ON EVERY RENDER ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ
  // No array = "run after every single render"
  useEffect(() => {
    console.log('This runs on EVERY render. Use sparingly!');
  });  // <-- No dependency array at all

  // ‚îÄ‚îÄ 4. CLEANUP (Unmount) ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ
  // The function you RETURN from useEffect is the cleanup
  useEffect(() => {
    const timer = setInterval(() => console.log('tick'), 1000);

    return () => {
      clearInterval(timer);  // This runs when component unmounts
      // Without this, the timer keeps running even after
      // the component is gone ‚Äî a memory leak!
    };
  }, []);
}

Real-World Lifecycle Example: Data Fetching

Here's a complete, practical example combining everything ‚Äî mounting, updating, and unmounting lifecycle:

components/UserProfile.jsx ‚Äî Full Lifecycle Example 
function UserProfile({ userId }) {
  const [user, setUser] = useState(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    // MOUNT + UPDATE: fetch user whenever userId changes
    setLoading(true);

    // AbortController lets us cancel the fetch on unmount
    const controller = new AbortController();

    fetch(`/api/users/${userId}`, { signal: controller.signal })
      .then(res => res.json())
      .then(data => {
        setUser(data);
        setLoading(false);
      })
      .catch(err => {
        if (err.name !== 'AbortError') setError(err);
      });

    // CLEANUP (Unmount): Cancel in-flight request
    return () => controller.abort();

  }, [userId]);  // Re-run whenever userId changes

  if (loading) return <p>Loading...</p>;
  if (error)   return <p>Error: {error.message}</p>;

  return <div>{user.name}</div>;
}
The Dependency Array Mental Model

Think of the dependency array as a watch list. You're saying: "React, watch these values. Whenever any of them change, re-run my effect." An empty [] means "watch nothing ‚Äî run once." No array means "run after everything." Include every variable your effect uses in the array.

State ‚Äî The Brain of React

State is the most powerful and important concept in React. If you truly understand state, you understand React. Let's go deep.

"State is data that, when changed, causes your component to re-render with the new data automatically."

Why Not Just Use Regular Variables?

This is the #1 question beginners have. Let's prove why regular variables don't work:

‚ùå Why Regular Variables Fail 
function Counter() {
  let count = 0;  // Regular variable

  return (
    <div>
      <p>Count: {count}</p>
      <button onClick={() => count++}>Increment</button>
    </div>
  );
}
// üö® PROBLEM: count DOES increment internally,
// but React has NO IDEA it changed!
// React only re-renders when STATE changes.
// Regular variables silently change and nothing updates on screen.
‚úÖ useState ‚Äî The Right Way 
import { useState } from 'react';

function Counter() {
  // useState returns [currentValue, setterFunction]
  // 0 is the initial value
  const [count, setCount] = useState(0);

  return (
    <div>
      <p>Count: {count}</p>
      <button onClick={() => setCount(count + 1)}>
        Increment
      </button>
    </div>
  );
  // ‚úÖ When setCount is called:
  // 1. React saves the new count value
  // 2. React re-renders the Counter component
  // 3. The new count appears on screen
  // Automatic. Magical. Reliable.
}

State with Objects and Arrays

When state is an object or array, you must not mutate it directly. You must create a new copy with the changes. This is how React detects that something changed:

State with Objects & Arrays 
function ProfileEditor() {
  const [user, setUser] = useState({
    name: 'Alice',
    age: 28,
    email: 'alice@example.com'
  });

  const [hobbies, setHobbies] = useState(['coding', 'hiking']);

  // ‚ùå WRONG ‚Äî direct mutation, React won't detect change!
  const badUpdate = () => {
    user.name = 'Bob';   // Mutates existing object
    setUser(user);        // Same reference = React ignores it
  };

  // ‚úÖ CORRECT ‚Äî spread creates a new object with changes
  const updateName = () => {
    setUser({ ...user, name: 'Bob' });
    // ...user copies all existing properties
    // name: 'Bob' then overrides just the name
  };

  // ‚úÖ CORRECT ‚Äî adding to an array
  const addHobby = () => {
    setHobbies([...hobbies, 'painting']);
    // Spread existing + add new item at end
  };

  // ‚úÖ CORRECT ‚Äî removing from array
  const removeHobby = (hobbyToRemove) => {
    setHobbies(hobbies.filter(h => h !== hobbyToRemove));
    // .filter creates a NEW array without the removed item
  };
}

The Functional Update Pattern

When new state depends on old state, always use the functional form of the setter. This is critical when multiple state updates happen in quick succession:

Functional Updates ‚Äî Critical Pattern 
const [count, setCount] = useState(0);

// ‚ùå RISKY ‚Äî may use a stale count value
const badIncrement = () => setCount(count + 1);

// ‚úÖ SAFE ‚Äî prev is ALWAYS the latest value
// React passes you the actual current state as "prev"
const safeIncrement = () => setCount(prev => prev + 1);

// This matters when you call setCount multiple times fast:
const tripleIncrement = () => {
  setCount(prev => prev + 1);  // prev = 0, returns 1
  setCount(prev => prev + 1);  // prev = 1, returns 2
  setCount(prev => prev + 1);  // prev = 2, returns 3
  // Result: count = 3 ‚úÖ (correct!)

  // vs the BAD way:
  setCount(count + 1);  // count is 0, sets to 1
  setCount(count + 1);  // count is STILL 0, sets to 1 again
  setCount(count + 1);  // count is STILL 0, sets to 1 again!
  // Result: count = 1 ‚ùå (wrong!)
};

Lifting State Up ‚Äî Sharing Between Components

When two sibling components need to share the same state, you lift the state up to their common parent. The parent owns the state and passes it down as props:

Lifting State Up Pattern 
// Parent owns the state ‚Äî it's the single source of truth
function TemperatureApp() {
  const [celsius, setCelsius] = useState(0);

  return (
    <div>
      {/* Both children get state + the setter function */}
      <CelsiusInput
        value={celsius}
        onChange={setCelsius}
      />
      <FahrenheitDisplay
        value={celsius * 9/5 + 32}
      />
    </div>
  );
}

// Child 1 ‚Äî can change state via the onChange prop
function CelsiusInput({ value, onChange }) {
  return <input value={value} onChange={e => onChange(Number(e.target.value))} />;
}

// Child 2 ‚Äî reads derived state, doesn't modify anything
function FahrenheitDisplay({ value }) {
  return <p>{value}¬∞F</p>;
}
// When CelsiusInput changes, Parent re-renders,
// FahrenheitDisplay automatically shows the updated conversion!
State Summary ‚Äî The 5 Rules

1. Never mutate state directly ‚Äî always use the setter function
2. Objects/arrays need new copies ‚Äî use spread { ...obj } or [...arr]
3. Use functional updates when new state depends on old state
4. State is local by default ‚Äî lift up to share between siblings
5. State updates may be batched ‚Äî React can group multiple setters into one re-render for performance
